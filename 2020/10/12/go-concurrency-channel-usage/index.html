Go concurrency and Channel

· 2020-10-12 ·次阅读

Go并发与Channel

1.Channel创建和关闭

//interface{}是该通道的类型，也可以是别的int、string、自定义结构体类型等等。甚至还可以时通道本身。
var dataStream chan interface{}
dataStream = make (chan interface{})

//直接用make建立
dataStream := make(chan interface{})

//关闭通道
close(dataStream)

2.Channnel的读写和缓冲

//向通道写入
dataStream<- 5
//从通道中读取
<-dataStream
i,ok := <-dataStream//第二个参数读取通道开关状态

//有缓冲通道,缓冲区大小为4
dataStream := make(chan int，4）

//单向通道
dataStream := make(<-chan int)
dataStream := make(chan<- int)
//一般用于约数channel写入或读取范围，多用于参数传递

Channel的读写

  channel有开启和关闭两种状态，通道开启时，读取元素个数为0的通道会发生阻塞。通道关闭时，仍然可以从通道中读取剩余的元素，当所有元素都读取完后，就会读取这个channel类型的零值。如果时int，读取0；如果是string，读取””。

  需要注意的是创建channel没有调用make，将会是一个nil的channel。不管是读取还是写入，都会造成死锁。

fatal error: all goroutines are asleep - deadlock!
goroutine 1 [chan receive (nil chan)]:

fatal error: all goroutines are asleep - deadlock!
goroutine 1 [chan send (nil chan)]:

  对一个只能写入的通道读取或者对一个只能读取的通道写入都会报错

invalid operation: stream <- 4 (send to receive-only type <-chan int)

单向通道用法：

  多用于参数传递

in := func() <-chan int {
  stream := make(chan int)
  go func() {
    defer close(stream)
    stream<- 5
   }()
   return stream
}

  像这个叫做in的函数，它返回了一个只读通道，规定了函数外只能读取。可以防止channel的滥用。

一个程序案例

func chanOwner() <-chan int {
  resultStream := make(chan int,6)//1
  go func() {
    defer close(resultStream)
    for i := 0; i<=5; i ++ {
      resultStream<- i
    }
  }()
  return resultStream
}

func main() {
  resultStream := chanOwner()
  for {
    i,ok := <-resultStream
    if !ok {
      break
    }
  }
}

  可以想一下把1位置的6换成4会发生什么。

  想想为什么关闭了goroutine里面的resultStream,外面的也被关闭了

3.并发的错误处理

  错误处理是程序中重要的一环，程序运行时会产生很多错误。像API没有获取到请求的body、mysql没有查询到相应记录、爬虫发送请求没有收到响应、调用包的函数也会有错误返回等等。对于这些产生的错误，只是把它们打印出来是不够的。在一般函数中，我们选择return 错误。将错误返回给上级函数来处理。因为上级函数可以更好的联系上下文，交给他们来处理会更好。那么问题来了，按照一般函数的形式来处理并发好像……并……不那么可行。

add := func(i *int) error {
  *i ++
  if *i == 2 {
    return nil
  }
  return errors.New("wrong")
}

//正常函数的错误返回
err := add(&i) 
//some action due with err here

//并发处理是这样呢？
go err:= add(&i)

//还是这样呢？
err := go add(&i)

  如果你真的写出了上面两个代码，那么编译器就会送两句话给你：

expression in go must be function call
syntax error: unexpected go, expecting expression

  很明显是不可行的，并发的错误处理，还是要靠channel

  我们只需要创建一个类型为error的channel就能解决错误传递问题

errStream := make(chan error)

add := func(i *int) {
  defer close(errStream)
  *i++
  if *i == 2 {
    errStream <- nil
  }
}

go add(&i)

err := <-errStream

  像这样就能得到goroutine中的错误。如果有多个goroutine，结果可以用range遍历处理。有时候我们goroutine对一些数据进行处理，会有一些数据需要我们返回。这个时候，我们其实可以和error一起返回，自定义一个结构体：

type results struct {
  Sum int
  err error
}

resultStream := make(chan result)

add := func(i int) {
  defer close(resultStream)
  i++
  if i == 2 {
    result:= results{Sum:i,err:nil}
    resultStream<- result
  }
  result:= results{sum:0,err:errors.New("wrong")}
  resultStream<- result
}

更具体的错误

  我们的错误有时候不仅仅只有一句话，我们需要更具体的信息。比如时间、哪个goroutine、原始错误和自定义的错误，甚至还能追踪栈堆。和上面同理，我们使用结构体来自定义一个错误类型：

type MyError struct {
    Inner error
    Message string
    StackTrace string
    Misc map[string]interface{}
}

  inner字段用来存储原始错误，message用来存储自定义错误信息，StackTrace用来追踪栈堆，misc也是自定义的，可以存储一些对找到错误有帮助的信息。

  接下来是对错误的写入：

func wrapError(err error, messagef string,msgArgs ...interface{}) MyError {
    return MyError{
        Inner: err,
        Message: fmt.Sprintf(messagef,msgArgs...),
        StackTrace: string(debug.Stack()),
        Misc: make(map[string]interface{}),
    }
}

func (err MyError) Error() string {
    return err.Message
}

  当要返回错误的时候，就调用wrapError函数写入。定义一个Error()方法方便提取自定义错误信息。

  然后是错误的输出：

func handleError(key int,err error,message string) {
    log.SetPrefix(fmt.Sprintf("[logID:%v]: ",key))
    log.Printf("%#v",err)
    fmt.Printf("[%v] %v", key,message)
}

4.防止goroutine泄漏

  goroutine有以下三种情况被终止

1.goroutine完成了自己的工作

2.goroutine遇到不可恢复的错误，不能工作

3.goroutine被告知终止工作

  goroutine也有以下几种方式泄露：

1.尝试读取或写入nil channel

func main() {
    doWork := func(strings <-chan string) <-chan interface {} {
        completed := make(chan interface{})
        go func() {
            defer fmt.Println("doWork exited.")
            defer close(completed)
            for s := range strings{
                //do something
                fmt.Println(s)
            }

        }()
        return completed
    }

    doWork(nil)
    //any other action
    fmt.Println("Done.")
}

  从示例看出，dowork函数被传进了一个nil。而后面的goroutine尝试对nil channel读取，使得整个goroutine无法关闭。在真正的程序中dowork传进的参数往往是上一步中的结果。上一步的结果不排除会出现错误而返回nil的情况。当在整个程序生命周期中存在无用的goroutine时，内存利用率会下降。

  要关闭这个通道，我们需要用到上述第三种方法。也就是发送消息给goroutine让它关闭。

func main(){
    doWork := func(done <-chan interface{}, strings <-chan string) <-chan interface{} {
        terminated := make(chan interface{})
        go func() {
            defer fmt.Println("doWork exited.")
            defer close(terminated)

            for{
                select {
                    case s := <-strings :
                        //do something
                        fmt.Println(s)
                    case <-done :
                        return
                }
            }
        }()

        return terminated
    }

    done := make(chan interface{})
    terminated := doWork(done, nil)

    go func() {
        //1s后发送关闭通知
        time.Sleep(1 * time.Second)
        fmt.Println("Cancelling doWork goroutine...")
        close(done)
    }()

    <-terminated
    fmt.Println("Done.")

  这里我们利用通道的性质，建立一个done channel就可以解决。done channel中没有元素，所以在goroutine中会阻塞，但是一旦把它关闭了，就可以从中读取元素，就可以关闭goroutine了。

2. 有发送者却没有接收者

func main(){
    newRandStream := func() <-chan int{
        randStream := make(chan int)
        go func() {
            defer fmt.Println("newRandStream closure exited.")
            defer close(randStream)
            for{
                randStream <- rand.Int()
            }
        }()
        return randStream
    }

    randStream := newRandStream()
    fmt.Println("3 random ints:")
    for i:=1; i<=3; i++{
        fmt.Printf("%d: %d\n", i, <-randStream)
}

  像上面这个例子，goroutine不断工作，将生成的随机数不断地写入channel（生成器），但是main goroutine只读取了三次就进入下一个模块了，而goroutine却不知道停下来。因此我们需要用一个done channel告诉它你工作做完了，该下班了。

func main(){
    newRandStream := func(done <-chan interface{}) <-chan int {
        randStream := make(chan int)
        go func() {
            defer fmt.Println("newRandStream closure exited. ")
            defer close(randStream)

            for{
                select {
                    case randStream <-rand.Int() :
                        case <-done :
                            return
                }
            }
        }()

        return randStream
    }

    done := make(chan interface{})
    randStream := newRandStream(done)
    fmt.Println(" 3 random ints:")
    for i:=1; i<=3; i++ {
        fmt.Printf("%d: %d\n", i, <-randStream)
    }
    close(done)

    //some other action follow
    time.Sleep(1 * time.Second)
}

  稍微修改以下程序就能解决。

3.有接收者但没有发送者

  这种情况和上面是类似的，可能出现在上一个goroutine已经工作完，但是这个goroutine不知道，仍然在接收。可能就是channel没有关闭。也是用done channel来解决。

另外有一个约定：如果goroutine 负责创建 goroutine,他也负责 确保可以停止 goroutine

5.其他channel的使用

1. or-channel / 复合done channel

  or-channel 可以将多个 done channel 合并为一个，并且当其中任意一个通道关闭时，合并的新通道就会关闭。（但是合并通道中没有关闭的通道依然没有关闭）

func main(){
    var or func(channels ...<-chan interface{}) <-chan interface{}

    or = func(channels ...<-chan interface{}) <-chan interface{}{
        switch len(channels){
        case 0 : 
            return nil
        case 1 :
            return channels[0]
        }

        orDone := make(chan interface{})

        //创建goroutine 以便可以不受阻塞地等待我们 channel上的消息
        go func(){
            defer close(orDone)
            switch len(channels){
            case 2 :
                select{
                    case <-channels[0] :
                    case <-channels[1]:
                }
            default :
                select{
                    case <- channels[0] :
                    case <- channels[1] :
                    case <- channels[2]    :
                    case <- or(append(channels[3:], orDone)...) : 
                    //为了使在建立这个树的goroutine 退出时，树下的goroutine 也可以跟着退出，需要将这个orDone channel 也传递到了调用中。
                }
            }

        }()

        return orDone
    }

    sig := func(after time.Duration) <-chan interface{} {
        c := make(chan interface{})
        go func(){
            defer fmt.Printf("done ! after %v",after)
            defer close(c)
            time.Sleep(after)
        }()
        return c
    }
    start := time.Now()
    <-or(
        sig(2*time.Hour),
        sig(2*time.Minute),
        sig(1*time.Second),
        sig(5*time.Second),
        sig(10*time.Second),
        sig(1*time.Hour),
        sig(1*time.Minute),
        )
    fmt.Printf("done after %v", time.Since(start))
    time.Sleep(5*time.Minute);
}

  合并过程中，不同的done通道需要不同的时间关闭，但是合并之后的新通道只需要1s就关闭了。可以看见or-channel是通过递归实现的。每次传递orDone可以保证使所有的goroutine退出。

  这种模式在系统中的模块交汇处非常有用。

2.tee-channel

  tee-channel能够把一个channel 分为两个（其实就是复制），这两个channel可以发送到代码中两个不同的区域中。

func main() {
    tee := func(done <-chan interface{}, in <- chan interface{})( <-chan interface{}, <-chan interface {}) {
        out1 := make(chan interface{})
        out2 := make(chan interface{})

        go func() {
            defer close(out1)
            defer close(out2)
            for val := range orDone(done, in){
                //两个通道重置
                var out1, out2 = out1, out2
                for i := 0; i < 2; i++{//可以改变通道的个数
                    select {
                    case <- done :
                    case out1<- val:
                        out1 = nil
                    case out2<- val:
                        out2 = nil
                    }
                }
            }

        }()
        return out1, out2
    }

    done :=make(chan interface{})
    defer close(done)
    out1, out2 := tee(done, take(done, repeat(done,1,2), 4))//调用了一个生成器

    for val1 := range out1{
        fmt.Printf("out1:%v, out2:%v\n", val1, <-out2)
    }
}

  tee-channel实际上就是复制通道。当channel放进了一个值，并且我们多个goroutine需要用到它时，就可以调用tee channel

3.or-done-channel

  我们程序可能会需要处理大量的、且来自程序不同模块的channel。但是我们处理这些channel时，不知道发送这些channel的goroutine是否关闭。（现在是goroutine未关闭才读取。我们上面的例子的channel都没有缓冲，都是一边写入一边读取。）这个时候就需要选择正确的channel（那些没有退出的channel）的进行接收。那么判断goroutine是否关闭其实就是判断channel是否关闭（因为goroutine都有defer close（），如果没有defer说明你goroutine有问题）。而or-done-channel就是干这行的：

orDone := func(done,c <-chan interface{}) <-chan interface{} {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            select {
                case <-done:
                    return
                case v,ok := <-c://判断是否goroutine是否关闭
                    if  ok == false {
                        return
                    }
                    select {
                        case valStream <-v: //关闭才写入返回流
                        case <-done:
                    }
            }
        }
    }()

    return valStream
}

//调用就非常简单了：
for val := range orDone(done,myChan) {//调用如果不想用done，也可以代入nil  myChan是目标通道
    //do something
}

4.bridge-channel

  程序中可能会有很多channel给一个goroutine进行消费（处理channel的channel）。作为消费者，我们不需要知道消费的是谁的channel，干就完了。而bridge-channel就是将这么一个channel的channel拆解为一个channel的工具。

//下面就是channel的channel类型
<-channel <-channel interface{}

bridge := func(done <-chan interface{},chanStream <-chan <-chan interface{}) <-chan interface {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            var stream <-chan interface{}
            select {
                case maybeStream,ok := <-chanStream :
                    if ok == false {//关闭的通道不要合并
                        return
                    }
                    stream = maybeStream
                case <-done:
                    return
            }
            for val := range orDone(done,stream) {//这里调用了上面提到的or-done-channel来判断是否channel是否可处理
                select {
                    case valStream <-val:
                    case <-done:
                }
            }
        }
    }()
    return valStream
}

//调用也非常简单
for v := range bridge(nil,[...Your Channel...]) {
    //do something
}
